# Contract: 语音管线（流式 STT → Claude 流式文本 → 流式 TTS）

**Feature**: [../spec.md](../spec.md) | **Research**: [../research.md](../research.md) §1, §2, §8

本契约定义 Strategy Service 的 `voice/` 模块内部各阶段之间的数据流与延迟预算分配，用于约束实现并作为 SC-005（<500ms 语音交互）的可验证依据。总预算 500ms 需要在阶段间明确分配，避免"到实现时才发现预算不够"。

## 阶段与延迟预算

```
玩家说话
   │
   ▼ (玩家端 player-voice-client 通过 WebRTC 实时上行音频流)
┌─────────────────┐
│ 1. 流式 STT       │  目标：识别到语句结束后 ≤ 200ms 内产出最终转写
│    （增量部分结果）│  （对应 PRD 原始非功能需求"语音转文本延迟<200ms"）
└────────┬────────┘
         │ transcript（可能为增量，供 Claude 提前预热但不提前提交完整请求）
         ▼
┌─────────────────┐
│ 2. Claude 流式文本 │  目标：首个文本 token ≤ 200ms（Sonnet 5, effort:low,
│    生成 + 工具调用  │  流式输出）；工具调用（如需要）与文本可并行返回
└────────┬────────┘
         │ 文本以句子/短语为单位分段（非等待整段生成完毕）
         ▼
┌─────────────────┐
│ 3. 流式 TTS       │  目标：收到第一个可合成片段后首字节 ≤ 300ms
│    （逐段合成播放） │  （对应 PRD 原始非功能需求"文本转语音首字延迟<300ms"）
└────────┬────────┘
         │ audio_chunk（流式下发给 Bot Client / player-voice-client 播放）
         ▼
玩家听到 AI 队友开始说话

总预算：从"玩家说完话"到"AI 队友开始可听见的回应"合计 < 500ms（FR-011/SC-005）。
```

**关键设计约束**：第 2、3 阶段必须**流水线重叠**而非串行等待——TTS 不等 Claude 生成完整回复才开始，而是消费 Claude 流式输出的文本片段（如按句子边界切分）逐段合成。若严格串行执行三阶段的目标延迟之和，理论上限已经等于总预算，没有余量应对网络抖动，因此流水线重叠是达标的必要条件，不是可选优化。

## 打断处理（FR-010）

玩家可在 AI 队友说话的任意时刻打断：
1. 流式 STT 检测到玩家开始说话（哪怕 AI 仍在播放语音）。
2. player-voice-client / Bot Client 立即发送 `voice.interrupt`（见 `websocket-protocol.md`），Strategy Service 收到后立即终止当前 TTS 流的下发与 Claude 生成请求（若仍在进行）。
3. 新一轮 STT → Claude → TTS 流程基于玩家的新输入重新开始。
4. 该次 Voice Exchange 记录 `interrupted = true`（见 data-model.md）。

## 身份披露与内容边界

- Claude 生成的 `chat_say` 文本需遵循 FR-022 的折中披露策略（认真询问如实告知）与 FR-021 的敏感话题转移——这些是 Prompt/系统提示层面的约束，不在本协议的传输层处理，但语音管线本身**不得**对文本内容做任何可能绕过该策略的后处理（如缓存/复用旧回复而不经过每次生成的策略判断）。

## 隐私处理

- 语音管线不持久化原始音频（FR-025）；`transcript` / `response_text` 若因 FR-018（玩家偏好）需要跨会话留存，只保留文本摘要，不保留原始声纹特征。
- `processing_mode = local` 时（FR-026），STT/TTS 阶段需切换到玩家本地设备侧的轻量实现——具体本地模型选型不在 V1 阻塞范围内（research.md 未解决事项），但协议层的分段流水线设计需保持一致，便于云端/本地实现互换。
