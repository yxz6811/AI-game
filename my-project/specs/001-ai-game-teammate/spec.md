## User Scenarios & Testing *(mandatory)*

本规格描述的黑客松交付分为四个层级，层级之间存在**严格的 Stage Gate 依赖**：只有第一层（保底）通过 Gate 1，才允许投入第二层；只有第二层至少一个方向（语音或 DST）形成稳定 Demo，才允许正式接入第三层；第四层优先级最低且完全可选。技术基线为 [Project AIRI](https://github.com/moeru-ai/airi) 主仓库及其现有 Minecraft 服务——第一层的目标不是新增产品功能，而是把 AIRI 已有能力部署、配置、连接起来并验证可重复演示。以下六个用户故事对应 PRD2 的四个层级（第二层拆为两个独立故事，第三层拆为两个并列可选故事，第四层的三个子方向合并为一个"任选其一"的故事），按优先级排列，每个故事均可独立验证、独立交付价值，且高优先级故事失败或未完成时不影响已完成的低编号故事继续运行。

### User Story 1 - 保底交付：Project AIRI × Minecraft 全链路打通 (Priority: P1) 🎯 MVP

部署、配置并连接 Project AIRI 现有模块到指定的 Minecraft 服务器，**不新增任何产品功能**，只验证"玩家输入 → AIRI 理解与决策 → Minecraft 内行动 → 游戏状态/事件反馈 → AIRI 产生玩家可感知的回应"这一完整闭环，并固化为一段可重复的 3-5 分钟现场演示脚本。

**Why this priority**：这是整个项目能否交付的地基——第一层未通过 Gate 1，后续所有层级的开发都必须停止（对应 PRD2 §10.2 全局停止条件）；这也是唯一一层被明确标注为"必须"（Must Have）的交付。

**Independent Test**：从冷启动开始，在锁定版本的 AIRI 与 Minecraft 服务上，安排一名玩家独立按固定演示脚本连续跑三次完整流程（问候/指令响应/简单交互/成功事件/失败事件），全程无阻断级崩溃，即可判定该故事独立交付价值。

**Acceptance Scenarios**:

1. **Given** 已锁定 AIRI 版本/commit 与依赖环境，**When** 从冷启动执行启动命令，**Then** 目标机器可稳定进入 AIRI 界面/服务。
2. **Given** 已准备好受控 Minecraft 服务器与测试世界，**When** 真人玩家与 AIRI Bot 分别登录，**Then** 两者可进入同一世界。
3. **Given** 已按 AIRI 现有 Minecraft 服务路径启动并连通核心服务，**When** 服务启动完成，**Then** AIRI 能识别 Bot 在线并接收到基本游戏状态。
4. **Given** 玩家使用 AIRI 现有输入方式（文本或现有语音能力）发出简单意图（如"跟我来""到这里""停止"），**When** AIRI 完成理解与决策，**Then** Bot 在 Minecraft 中执行跟随、停止、移动、交互中至少三类基础行动。
5. **Given** 游戏中发生一次成功事件与一次失败/不可执行事件，**When** 事件回传到 AIRI，**Then** AIRI 产生玩家可感知的合理回应。
6. **Given** 已固化 3-5 分钟演示脚本（限定地图、任务、指令，减少开放世界随机性），**When** 连续执行三次，**Then** 每次均无阻断级崩溃。
7. **Given** 演示过程中出现常见故障，**When** 参照故障排查 Runbook 操作，**Then** 可在 2 分钟内恢复到可演示状态。

### User Story 2 - 全双工实时语音打断与重规划 (Priority: P2)

玩家与 AI 可同时说话；玩家能在 AI 发言或执行游戏任务期间的任意时刻打断，系统能立即停止旧语音播放、取消或重规划正在执行的游戏任务，并对玩家的新输入作出响应。

**Why this priority**：这是黑客松阶段最能体现"AI 队友像真人一样自然协作"差异化竞争力的增强，直接对应 Gate 2A；它建立在第一层稳定基线之上，但不影响第一层的保底交付（对应 PRD2 §5.4、§10.2：第二层任一模块关闭后第一层仍可独立演示）。

**Independent Test**：在第一层稳定基线上，独立测试语音打断成功率（20 次测试 ≥18 次正确停止旧语音）与动作取消成功率（10 次"停止/改派"测试 ≥8 次正确取消），且不影响第一层原有演示脚本，即可判定该故事独立交付价值，无需依赖 User Story 3。

**Acceptance Scenarios**:

1. **Given** 已接入持续麦克风音频流与语音活动检测（VAD），**When** 连续对话进行中，**Then** 系统可区分说话、静音与噪声状态。
2. **Given** AI 正在播放语音回复，**When** 玩家开口说话进行打断，**Then** 系统在 20 次测试中至少 18 次正确停止旧语音并开始处理玩家的新输入。
3. **Given** AI 正在执行一个游戏任务，**When** 玩家用语音说"等等/停下/回来"，**Then** 系统在 10 次"停止/改派"测试中至少 8 次正确取消旧任务并执行新指令。
4. **Given** 连续 10 分钟的语音对话，**When** AI 自身的语音输出正在播放，**Then** 系统不会将 AI 自己的 TTS 再次识别为玩家输入，无明显自激对话（自己跟自己说话）现象。
5. **Given** 一次响应出现明显延迟，**When** 查看延迟看板或结构化日志，**Then** 可定位该次慢响应具体发生在 VAD、STT、LLM、TTS、游戏行动中的哪个阶段。

### User Story 3 - 统一 Game Adapter 与 DST 第二游戏迁移 (Priority: P3)

定义统一的游戏状态/动作契约（`observe`/`act`/`events`/`capabilities`/`health`），将第一层已有的 Minecraft 链路包裹进该契约而不破坏其可运行基线，并新增 Don't Starve Together（饥荒联机版）适配器，验证同一套高层意图（跟随/移动/采集/交互/停止/说话）能在两款游戏中复用同一个 Agent 编排层。

**Why this priority**：证明"同一套 Agent 编排层可以跨游戏迁移"这一架构价值，直接对应 Gate 2B；是第二层增强的另一半，与 User Story 2（语音）相互独立，团队可并行推进或按成功概率优先其中一项（对应 PRD2 §5.2、§2.2）。

**Independent Test**：使用同一组高层意图分别在 Minecraft 与 DST 上执行跟随/停止/采集，验证核心编排逻辑没有被复制（仅 Adapter/Bridge 层不同），且运行第一层回归测试确认 Minecraft 原有演示仍可通过，即可判定该故事独立交付价值，无需依赖 User Story 2。

**Acceptance Scenarios**:

1. **Given** 已定义统一的游戏状态 Schema（角色位置、生命/饥饿/精神值、附近实体、资源、危险、任务状态），**When** 分别读取 Minecraft 与 DST 的角色状态，**Then** 两者均映射到同一顶层状态结构。
2. **Given** 已定义最小动作语义集（follow/move/collect/interact/stop/say），**When** 上层 Agent 发出动作指令，**Then** 无需感知底层按键操作或具体 Mod API 差异。
3. **Given** Minecraft Adapter 已将第一层链路包裹进统一契约，**When** 运行第一层回归测试，**Then** 第一层演示仍可正常通过，未被破坏。
4. **Given** 已通过专用服务器 Mod/本地桥接方式连接 Don't Starve Together，**When** AI 角色进入受控 DST 世界，**Then** 可执行最小动作集中的基础动作。
5. **Given** DST Adapter 已将状态和动作映射到统一契约，**When** 玩家发出"跟随/停止/收集"等同一意图，**Then** 该意图可在 DST 中被正确执行。
6. **Given** 同一组高层意图分别在两款游戏中测试，**When** 检查实现代码结构，**Then** 核心编排逻辑没有被复制，两款游戏仅在 Adapter/Bridge 层存在差异。

### User Story 4 - 游戏专用本地 SLM/SSM 反射模型 (Priority: P4)

基于人工整理、规则生成或已有游戏轨迹构造小规模监督数据，训练一个本地 Transformer SLM（小型语言/行动模型）或 SSM（状态空间模型）作为高频动作的"反射层"，在冻结测试集上验证动作正确率与决策延迟收益，并在遇到未知任务或低置信度输出时自动回退到 AIRI 基线。

**Why this priority**：属于第三层"超高收益可选增强"中的一条独立轨道，与 User Story 5（实体硬件桌宠）并列——团队可按能力择一或两者都做，不要求两者同时完成；此轨道失败不影响第一、二层交付（对应 PRD2 §6.1、§6.4）。

**Independent Test**：在固定 Demo 场景中现场对比 AIRI 基线与本地模型的动作选择、置信度、决策延迟，并现场断开模型服务验证主链路（第一、二层）是否继续正常运行，即可判定该故事独立交付价值，无需依赖 User Story 5 或 User Story 6。

**Acceptance Scenarios**:

1. **Given** 已限定 follow/stop/collect/interact/avoid/return 等高频高层动作边界，**When** 检查模型可输出的动作范围，**Then** 至少支持 5 类动作，且不承担开放式聊天任务。
2. **Given** 已整合人工标注、规则生成、教师模型标注或已有轨迹构建的数据集，**When** 检查数据集清单，**Then** 数据来源、版本与许可信息清晰，训练集与测试集无重复。
3. **Given** 已选择轻量 Transformer SLM 或 SSM 架构并完成训练（行为克隆/监督微调/策略蒸馏），**When** 在目标机器或单卡上运行，**Then** 可完成训练或加载既有训练结果。
4. **Given** 已在冻结测试集上运行离线评测，**When** 查看离线评测报告，**Then** 动作选择正确率达到 ≥90% 目标，且报告结果可复现。
5. **Given** 模型输出动作、参数与置信度，**When** 遇到未知任务、超时或低置信度输出，**Then** 系统自动回退到 AIRI 基线，且关闭模型服务后主链路继续运行。
6. **Given** 在固定 Demo 场景下人工重复测试，**When** 对比本地模型与 AIRI 基线，**Then** 决策延迟目标降低 ≥50%，且任务成功率无明显下降。

### User Story 5 - 实体硬件桌宠具身体验 (Priority: P5)

使用 ESP32 / Raspberry Pi 等最小硬件组合（屏幕、LED、舵机或扬声器）消费统一的体验事件（如 speaking/thinking/action/success/danger/hurt/death），将 AI 队友的说话、思考、行动、危险、受击、成功状态转化为实体表情、灯光、动作或声音，并在断电、断网或进程退出时不影响主链路继续运行。

**Why this priority**：属于第三层"超高收益可选增强"中与 User Story 4 并列的另一条独立轨道，能以较低的架构侵入换取强烈的现场记忆点与传播效果；此轨道失败不影响第一、二层交付，可用软件状态卡降级替代（对应 PRD2 §6.1、§6.3、§6.4）。

**Independent Test**：触发至少 5 类体验事件并验证肉眼可区分的表情/灯光/动作/声音反馈，随后现场拔电或断网，验证主游戏与语音链路（第一、二层）不受影响，即可判定该故事独立交付价值，无需依赖 User Story 4 或 User Story 6。

**Acceptance Scenarios**:

1. **Given** 已选定硬件形态并组合必要部件，**When** 设备上电，**Then** 可独立进入待机状态。
2. **Given** 已定义统一的体验事件 Schema，**When** Mock 环境、实体桌宠、软件降级方案分别消费事件，**Then** 三者使用同一套协议。
3. **Given** 桌宠通过 WebSocket / MQTT / 串口消费事件，**When** 连续事件持续到达，**Then** 不出现明显卡死或状态混乱。
4. **Given** 已定义事件到表情/灯光/动作/口型/音效的映射表，**When** 触发不同类别的事件，**Then** 至少 5 类状态在肉眼观察下可清晰区分。
5. **Given** AI 正在说话，**When** 语音播放进行中，**Then** 口型/灯效变化与音频的开始/结束基本保持同步。
6. **Given** 桌宠作为独立进程运行，**When** 拔电、断网或关闭该进程，**Then** 主 Demo（第一、二层）继续正常运行，不受影响。

### User Story 6 - 研究型增强验证：Agent Arena / Replay-to-Skill / Shadow Observer（任选其一）(Priority: P6)

在第三层至少一条主线稳定后，视剩余时间与团队资源，从以下三个研究型方向中选择至少一项，形成可重复证据或受控演示：(a) **Agent Arena** 自动化评测——标准任务的多轮自动运行与指标报告；(b) **Replay-to-Skill**——从一条成功轨迹生成可复用、可取消的行为技能；(c) **Shadow Observer**——纯视觉的影子世界模型，在没有结构化接口时只读补充事件。

**Why this priority**：优先级最低的前沿研究层，完全可选，不得成为前三层的运行依赖；任一子方向失败或未完成都不影响前三层的交付判断（对应 PRD2 §7.5、§10.2）。

**Independent Test**：任选三个子方向之一，按其自身验收标准独立验证（Arena：3 个任务各自动运行 10 轮并生成报告；Replay-to-Skill：生成的技能在不同初始位置连续运行 3 次至少成功 2 次；Shadow Observer：固定场景下三类视觉事件识别准确率 ≥80%），且关闭该模块后前三层运行不受影响，即可判定该故事独立交付价值。

**Acceptance Scenarios**:

**子方向 a：Agent Arena 自动评测**

1. **Given** 已建立至少 3 个可重复的标准场景（如 follow、collect、stop/danger），**When** 运行 Arena Runner，**Then** 每个场景可自动脚本化初始化、复位，并连续自动运行 10 轮，生成成功率、耗时、动作数、调用数、延迟与失败归因报告。
2. **Given** 已保存 AIRI 基线的运行结果，**When** 加入第三层本地模型进行对比，**Then** 同一版本、同一任务的结果可重复比较。

**子方向 b：Replay-to-Skill**

3. **Given** 一条已确认成功的轨迹（来自 Arena 或人工录制），**When** 系统将其抽象为参数化技能，**Then** 生成的技能包含进入、成功、超时、失败、取消五种分支，且可在不同初始位置复用。
4. **Given** 新生成的技能已经过人工确认与沙箱环境测试，**When** 在不同初始位置连续运行 3 次，**Then** 至少成功 2 次，且玩家发出"停止"指令时可中断该技能的执行。

**子方向 c：Shadow Observer 视觉影子世界模型**

5. **Given** 已定义至少 3 类视觉事件（如低血量、附近危险、目标资源/UI 完成），**When** Observer Service 处理采集到的画面，**Then** 从画面到事件产生的延迟不超过 1 秒。
6. **Given** 视觉事件与 Game Adapter 的结构化状态进行对照，**When** 在预设场景下测试，**Then** 三类事件的识别准确率达到 ≥80% 目标，且关闭视觉服务后游戏任务执行不受影响（视觉事件仅为只读补充，不直接控制角色）。

### Edge Cases

- 当 AIRI 版本或依赖发生漂移，导致无法启动或接口发生变化时，应如何降级（回退到已验证版本，而非追逐最新 main 分支）？
- 当 Minecraft 协议或账号出现问题导致 Bot 无法入服时，应如何切换到本地备用演示服务器？
- 当 LLM 在开放式任务中表现不稳定时，关键动作应如何由技能函数/状态机兜底执行，而非完全依赖模型逐帧控制？
- 当语音回声或打断机制失效，导致 AI 出现自说自话、体验混乱时，应如何关闭全双工能力并回退到第一层的原有输入方式？
- 当 DST Mod/Bridge 不稳定导致第二游戏无法正常演示时，应如何降级为仅展示 Adapter Mock 与 Minecraft 回归测试？
- 当训练数据不足或存在数据泄漏，导致本地小模型指标失真（过拟合）时，应如何处理（冻结数据版本、按场景划分、使用独立测试集）？
- 当本地 SLM/SSM 相比 AIRI 基线没有速度或稳定性收益时，是否应从主演示中移除，仅保留训练结果说明？
- 当硬件桌宠出现故障（无响应、状态错乱、拖垮主链路）时，应如何隔离故障并切换到软件状态卡或预录屏幕？
- 当 Agent Arena 的场景无法脚本化复位，导致实验结果不可比较时，应如何降级为展示离线轨迹回放与报告？
- 当 Replay-to-Skill 生成的技能存在不安全或不稳定行为时，应如何通过人工确认、沙箱、超时与取消机制拦截，避免污染主世界？
- 当 Shadow Observer 出现视觉误报或"幻觉"（识别到不存在的事件）时，应如何通过置信度阈值与结构化状态对照进行校验，或直接关闭该视觉事件消费者？
- 当范围膨胀（临时想法不断涌入）威胁核心链路交付进度时，应如何强制要求新想法只进入第三/四层 Backlog，而不反向污染已冻结的第一层？
- 当第一层已通过 Gate 1 后，又被后续跨层修改破坏（回归测试失败）时，应立即回滚到已验证版本，还是继续在现场版本上调试？
- 当距离最终演示的剩余时间不足总周期的 20% 时，是否应立即停止所有新增功能开发，只做稳定化、录屏与讲解材料准备？

## Requirements *(mandatory)*

### Functional Requirements

**第一层 · Project AIRI × Minecraft 全链路（对应 User Story 1）**

- **FR-001**: 系统 MUST 支持锁定 Project AIRI 的特定版本/commit 并完成依赖安装，使目标机器可从冷启动进入可用状态。
- **FR-002**: 系统 MUST 准备受控的 Minecraft 服务器与测试世界，确保真人玩家与 AI 队友的 Bot 可进入同一世界。
- **FR-003**: 系统 MUST 按 Project AIRI 现有 Minecraft 服务的组织方式启动并连通核心服务，使 AIRI 能识别 Bot 在线状态并接收基本游戏状态。
- **FR-004**: 系统 MUST 支持玩家使用 AIRI 现有输入方式（文本或现有语音能力）发出简单意图，并驱动 Bot 在跟随、停止、移动、交互中至少执行三类基础动作。
- **FR-005**: 系统 MUST 将游戏内的成功、失败或环境变化事件回传至 AIRI，并产生玩家可感知的合理回应。
- **FR-006**: 系统 MUST 固化一段限定地图、任务与指令范围、时长 3-5 分钟的演示脚本，并能连续执行三次而无阻断级崩溃。
- **FR-007**: 系统 MUST 提供故障排查 Runbook，使非核心开发者可在常见故障发生后 2 分钟内恢复到可演示状态。
- **FR-008**: 第一层 MUST NOT 引入全双工语音、持续监听/随时打断、第二款游戏、通用 Game Adapter 抽象、模型训练、Agent Arena、长期记忆、语音克隆、自定义人格编辑器、实体硬件、直播推流、移动端/主机支持，或任何动作游戏级（如 100ms）延迟承诺——这些均属于后续层级的范围。

**第二层 A · 全双工实时语音（对应 User Story 2）**

- **FR-009**: 系统 MUST 接入持续麦克风音频流与语音活动检测（VAD），能够区分说话、静音与噪声状态。
- **FR-010**: 系统 MUST 使用流式语音识别提供增量文本，且文本转语音 MUST 支持分段播放，使系统无需等待完整句子结束即可开始理解与回应。
- **FR-011**: 系统 MUST 在玩家开口说话时立即停止当前语音播放，并向 Agent 发送打断事件；20 次测试中 MUST 至少 18 次正确停止旧语音。
- **FR-012**: 系统 MUST 支持语音打断同时取消或重规划正在执行的游戏任务；10 次"停止/改派"测试中 MUST 至少 8 次正确取消旧动作并执行新指令。
- **FR-013**: 系统 MUST 通过回声消除/输出抑制策略避免将 AI 自身的语音输出再次识别为玩家输入；连续 10 分钟对话中 MUST NOT 出现明显自激对话现象。
- **FR-014**: 系统 MUST 记录 VAD、STT、LLM、TTS、游戏行动各阶段的处理时间，使团队能够定位一次慢响应具体发生在哪个阶段。

**第二层 B · 统一 Game Adapter 与 DST（对应 User Story 3）**

- **FR-015**: 系统 MUST 定义统一的游戏状态 Schema，抽取角色位置、生命/饥饿/精神值、附近实体、资源、危险与任务状态，使 Minecraft 与 DST 均映射到同一顶层状态结构。
- **FR-016**: 系统 MUST 定义统一的动作语义（至少覆盖 follow/move/collect/interact/stop/say），使上层 Agent 无需感知底层按键操作或具体 Mod API 差异。
- **FR-017**: 系统 MUST 提供 `observe()`、`act(action)`、`events()`、`capabilities()`、`health()` 五个标准接口，分别用于状态查询、动作执行、事件流输出、能力声明与连接健康状态报告。
- **FR-018**: 系统 MUST 将第一层已有的 Minecraft 链路包裹进统一 Game Adapter 契约，且不得破坏其原有可运行基线——第一层回归测试必须能够通过。
- **FR-019**: 系统 MUST 通过专用服务器 Mod 或本地桥接方式连接 Don't Starve Together，读取状态、发送动作与接收事件，使 AI 队友可进入受控 DST 世界并执行最小动作集。
- **FR-020**: 系统 MUST 将 DST 的状态与动作映射到统一契约，使跟随/停止/收集等同一高层意图可以在 DST 中被正确执行。
- **FR-021**: 系统 MUST 保证跨游戏的核心编排逻辑不被复制——Minecraft 与 DST 之间只应存在 Adapter/Bridge 层的实现差异。
- **FR-022**: 每个动作 SHOULD 具备超时、重试与取消机制，防止 Agent 长时间卡在不可达目标。
- **FR-023**: 适配器 MUST 通过 `capabilities()` 声明当前游戏支持的动作，对不支持的动作立即返回可解释的失败，而非静默挂起。

**第三层 A · 游戏专用本地 SLM/SSM（对应 User Story 4）**

- **FR-024**: 系统 MUST 将本地模型的动作边界限定为 follow/stop/collect/interact/avoid/return 等高频高层动作（至少 5 类），且 MUST NOT 承担开放式聊天。
- **FR-025**: 系统 MUST 使用人工标注、规则生成、教师模型标注或已有轨迹构建冻结的最小训练数据集，MUST 明确数据来源、版本与许可信息，训练集与测试集不得重复。
- **FR-026**: 系统 MUST 训练一个轻量 Transformer SLM 或 SSM（通过行为克隆、监督微调或策略蒸馏），并可在目标机器或单卡上完成训练，或加载既有训练结果。
- **FR-027**: 系统 MUST 在冻结测试集上生成可复现的离线评测报告，报告 MUST 包含动作正确率、推理延迟与错误类型。
- **FR-028**: 系统 MUST 让模型输出动作、参数与置信度，并在遇到未知任务或低置信度输出时自动回退至 AIRI 基线；关闭模型、超时或低置信度时主链路 MUST 继续运行。
- **FR-029**: 系统 MUST 在固定 Demo 场景下人工重复测试，比较本地模型与 AIRI 基线的成功率与延迟差异。

**第三层 B · 实体硬件桌宠（对应 User Story 5）**

- **FR-030**: 系统 MUST 选择 ESP32 或 Raspberry Pi 等硬件形态，组合屏幕、LED、舵机或扬声器中的必要部件，构建可独立上电进入待机状态的桌宠原型。
- **FR-031**: 系统 MUST 定义统一的体验事件 Schema（涵盖 speaking/thinking/action/success/danger/hurt/death 等及其优先级），供 Mock 环境、实体桌宠与软件降级方案共用同一协议。
- **FR-032**: 系统 MUST 通过 WebSocket / MQTT / 串口等方式消费体验事件，并处理节流、超时与状态覆盖，保证连续事件到达时不出现明显卡死或状态混乱。
- **FR-033**: 系统 MUST 将事件映射为屏幕表情、灯光、摆动、口型或音效，且至少 5 类状态 MUST 在肉眼观察下可清晰区分。
- **FR-034**: 系统 SHOULD 在 AI 说话时同步口型/灯效变化，或播放同一路 TTS 音频。
- **FR-035**: 桌宠 MUST 作为独立进程运行，支持断线重连与浏览器状态卡降级；MUST NOT 在拔电、断网或进程退出时影响主 Demo（第一、二层）的继续运行。
- **FR-036**: 实体桌宠 MUST NOT 读取模型内部思维链，也 MUST NOT 向游戏控制链路写入指令。

**第四层 · 研究型增强（对应 User Story 6，三选一）**

- **FR-037**: 若选择 Agent Arena 方向，系统 MUST 建立至少 3 个可重复的标准任务（如 follow、collect、stop/danger），支持脚本化初始化与自动复位，并能连续自动运行 10 轮，生成成功率/耗时/动作数/调用数/延迟/失败归因报告。
- **FR-038**: 若选择 Agent Arena 方向，系统 MUST 记录统一的轨迹 Schema（状态、指令、动作、事件、结果与各阶段延迟），支持单次运行回放关键决策证据；数据集 MUST 按场景/种子隔离并冻结版本，避免轨迹泄漏。
- **FR-039**: 若选择 Agent Arena 方向，系统 MUST 保存 AIRI 基线的运行结果，并可选加入第三层本地模型进行同版本、同任务的可重复对比。
- **FR-040**: 若选择 Replay-to-Skill 方向，系统 MUST 仅使用已确认成功的轨迹（来自 Arena 或人工录制）作为输入；失败或异常轨迹 MUST NOT 被自动发布为技能。
- **FR-041**: 若选择 Replay-to-Skill 方向，系统 MUST 将轨迹中的坐标与具体对象抽象为目标、条件、动作与终止条件，使生成的技能可在不同初始位置复用，并 MUST 包含进入、成功、超时、失败与取消五种分支。
- **FR-042**: 若选择 Replay-to-Skill 方向，新生成的技能 MUST 先在受控场景经人工确认与沙箱测试，未经确认 MUST NOT 用于主世界；技能 MUST 通过统一 Adapter 执行，并支持取消、超时与回退。
- **FR-043**: 若选择 Replay-to-Skill 方向，生成的技能在不同初始位置连续运行 3 次时，MUST 至少成功 2 次。
- **FR-044**: 若选择 Shadow Observer 方向，系统 MUST 以低帧率截图或视频流采集固定游戏窗口，且采集 MUST NOT 明显影响游戏帧率。
- **FR-045**: 若选择 Shadow Observer 方向，系统 MUST 定义至少 3 类视觉事件（如低血量、附近危险、目标资源/UI 完成），并通过现有或轻量检测模型输出事件与置信度；画面到事件产生的延迟 MUST 不超过 1 秒。
- **FR-046**: 若选择 Shadow Observer 方向，系统 MUST 将视觉事件与 Game Adapter 的结构化状态进行一致性/漏检/误报对照，并在预设场景下达到三类事件识别准确率 ≥80% 的目标。
- **FR-047**: 若选择 Shadow Observer 方向，视觉事件 MUST 仅作为只读输入触发语音提醒、桌宠反馈或 Arena 标记，MUST NOT 直接控制游戏角色；关闭视觉服务后，游戏任务执行 MUST NOT 受影响。

**跨层治理要求**

- **FR-048**: 系统 MUST 遵循 Stage Gate 纪律——只有 Gate 1（第一层可稳定演示）通过后，才能投入第二层开发；只有 Gate 2A 或 Gate 2B 至少有一项形成稳定 Demo 后，才能正式接入第三层。
- **FR-049**: 任何第二、三、四层模块关闭或运行失败时，第一层 MUST 能够独立继续运行和演示；第三、四层模块关闭时，其前序已完成的层级同样 MUST 不受影响。
- **FR-050**: 每次跨层合并代码 MUST 运行第一层回归脚本，且 MUST 保留可快速回滚的版本标记。
- **FR-051**: 系统 MUST NOT 提供任何绕过反作弊或破坏公平性的竞技类自动化能力。

### Key Entities *(include if feature involves data)*

- **AIRI 实例 (AIRI Instance)**：Project AIRI 的一次部署运行，持有锁定的版本/commit、模型提供商配置、人格设定与 Agent 编排逻辑；是第一层的核心承载对象。
- **游戏会话 (Game Session)**：一次有边界的玩家与 AI 队友共同游玩过程，关联具体游戏（Minecraft 或 DST）、当前任务/目标与连接健康状态。
- **Game Adapter**：把上层统一的状态/动作契约（observe/act/events/capabilities/health）映射到具体游戏实现的适配模块；Minecraft Adapter 与 DST Adapter 是其两个具体实现（第二层起存在）。
- **Bridge**：位于游戏侧或本地进程中的连接层，例如 Minecraft Bot 服务或 DST 服务端 Mod，负责与游戏本体的实际通信。
- **体验事件 (Experience Event)**：供实体桌宠、软件降级 Avatar、看板等消费的统一事件，携带类型（speaking/danger/success 等）与优先级（第三层起存在）。
- **动作模型输出 (Model Action Output)**：本地 SLM/SSM 针对一次决策请求输出的动作、参数与置信度，供置信度路由决定是否执行或回退 AIRI（第三层模型轨道存在）。
- **轨迹 (Trajectory)**：Agent Arena 单次运行记录的状态、指令、动作、事件、结果与各阶段延迟的完整序列（第四层 Arena 方向存在）。
- **技能 (Skill)**：Replay-to-Skill 从成功轨迹抽象出的参数化行为定义，包含进入/成功/超时/失败/取消分支（第四层 Replay 方向存在）。
- **视觉事件 (Vision Event)**：Shadow Observer 从画面识别输出的只读事件，携带类型与置信度（第四层视觉方向存在）。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**：（第一层，必须）"输入 → 理解 → 行动 → 反馈 → 回应"五段链路均有可观察证据。
- **SC-002**：（第一层，必须）预设演示脚本连续成功 3 次，无阻断级崩溃。
- **SC-003**：（第一层，必须）常见故障可在 2 分钟内按 Runbook 恢复到可演示状态。
- **SC-004**：（第二层，目标）20 次人工测试中语音打断成功率达到 ≥90%（即至少 18 次正确停止旧语音）。
- **SC-005**：（第二层，目标）10 次"停止/改派"测试中动作取消成功率达到 ≥80%（即至少 8 次正确取消）。
- **SC-006**：（第二层，必须）Minecraft 与 DST 两款游戏共享同一套上层意图、任务状态机与 Agent 编排逻辑。
- **SC-007**：（第三层，可选）SLM/SSM 在冻结测试集上的动作正确率目标 ≥90%；决策延迟相比 AIRI 基线降低目标 ≥50%，且任务成功率无明显下降。
- **SC-008**：（第三层，可选）实体桌宠至少 5 类事件在肉眼观察下可清晰区分；事件触发后 1 秒内可感知到设备状态变化。
- **SC-009**：（第三层，必须）模型服务或硬件桌宠关闭/断开后，第一、二层继续正常运行。
- **SC-010**：（第四层，可选）若选择 Agent Arena：至少 3 个标准任务，每个任务自动运行 10 轮并生成可比较报告。
- **SC-011**：（第四层，可选）若选择 Replay-to-Skill：生成的技能在不同初始位置连续运行 3 次，至少成功 2 次，且支持玩家指令取消。
- **SC-012**：（第四层，可选）若选择 Shadow Observer：固定场景下三类视觉事件识别准确率目标 ≥80%，画面到事件延迟目标 ≤1 秒。
- **SC-013**：（第四层，必须）任一已启用的第四层研究模块关闭后，第一、二、三层继续正常运行。

## Assumptions

- 项目形态为黑客松原型/技术验证，不等同于正式商业产品；不承诺 7×24 小时服务可用性、正式 SLA 或全平台覆盖。
- 技术基线为 Project AIRI 主仓库及其现有 Minecraft 服务；第一层不对 AIRI 做通用化重构，只按其当前主仓库与 `services/minecraft` 的真实组织方式接入。
- 第一层唯一游戏为 Minecraft；第二款游戏（Don't Starve Together）仅在第二层引入，且第二层不要求覆盖任意数量游戏，只需证明架构可迁移到一款新游戏。
- 第三层的 SLM/SSM 不从零预训练通用基础模型，仅允许基于人工整理、规则生成或已有游戏轨迹做小规模行为克隆、监督微调或策略蒸馏。
- 第三层模型训练不依赖第四层 Agent Arena 作为前置条件，可使用冻结的小规模数据集独立完成最小训练与离线验证；Arena 完成后可为其提供更可靠的自动化 A/B 评测，但不改变模型轨道可独立运行的事实。
- 第三层的模型轨道（User Story 4）与硬件轨道（User Story 5）相互独立、并列存在，团队可按自身能力择一或两者都做，不要求两者同时完成才算第三层达标。
- 第四层的三个研究方向（Agent Arena / Replay-to-Skill / Shadow Observer）任选其一形成可重复证据或受控演示即可，不要求全部完成。
- Live2D/VRM 形象、观众看板、人设包、局内短期记忆不再作为独立的层级交付目标，仅作为第三层实体桌宠的软件降级方案或配套展示 Backlog。
- 具体的 AIRI 版本/commit、Minecraft 服务器版本、模型提供商、目标演示机器与 Demo 世界，由执行阶段的"Phase 0：范围冻结与版本锁定"确认，不在本规格中预先指定固定值。
- 不做竞技游戏自动化，不做任何绕过反作弊或破坏公平性的能力；本约束贯穿全部四个层级，不因层级提升而放宽。
- 距离最终演示的剩余时间不足总执行周期的 20% 时，团队应停止一切新增功能开发，只做稳定化、回归验证、录屏与讲解材料准备。
