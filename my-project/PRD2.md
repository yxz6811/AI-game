## 0.1 对 V1.1 的 Review 与修订结论

- 第一层继续严格限定为 Project AIRI 与 Minecraft 的现有全链路部署、配置、连接和验证；本次不向第一层加入任何功能。
- 第二层继续只承载两项高收益增强：全双工实时语音，以及通过 DST 与统一 Game Adapter 证明跨游戏迁移。
- 第三层重构为两条独立的超高收益轨道：游戏专用本地 SLM/SSM，以及实体硬件桌宠。两者均可单独成立，失败时不影响第一、二层。
- 实体硬件桌宠从原第四层前移至第三层，因为它能以较低架构侵入换取极强的现场记忆点和传播效果。
- 第四层重构为研究型增强层，只承载 Agent Arena 自动评测、Replay-to-Skill 和视觉影子世界模型。
- 第三层模型可使用人工整理、规则生成或现有轨迹数据完成最小训练与离线测试，不把第四层 Agent Arena 设为前置依赖；Arena 完成后可进一步升级数据与评测质量。
- Live2D / VRM、观众看板、人设包和局内短期记忆不再作为独立层级目标，仅作为第三层桌宠的软件降级或配套展示 Backlog。
- 第三、四层均为完全可选；任何模块失败或关闭，不得影响第一、二层运行。

> **第一层硬边界：** 第一层只允许部署、配置、连接和验证 Project AIRI 现有 Minecraft 链路。不得在第一层加入全双工语音、第二游戏、模型训练、评测场、技能学习、视觉模型、长期记忆、硬件或通用插件框架。必要日志、配置文件和启动脚本属于交付保障，不属于新增产品功能。

## 0.2 四层关系与资源纪律

- 只有 Gate 1 通过后，才能投入第二层；只有 Gate 2A / 2B 至少形成稳定 Demo 后，才能正式接入第三层。
- 第三层的 SLM/SSM 与实体桌宠为并列轨道，可按团队能力择一；模型必须有冻结数据与最小离线测试，硬件必须通过断线隔离测试。
- 第四层优先级低于第三层，且不得成为第三层 SLM/SSM 或桌宠的运行依赖。
- 第四层的 Agent Arena 可反向为第三层模型提供更好的自动评测，但模型必须在没有 Arena 时仍可离线验证和一键回退。
- 任何临时想法先进入第三或第四层 Backlog，不得反向污染第一层。
- 最终演示前剩余时间不足总周期 20% 时，停止新增功能，只做稳定化、录屏和讲解材料。

# 1. 产品目标与演示命题

## 1.1 产品愿景

验证一种“AI 不只聊天，而是能作为队友进入游戏世界、理解玩家意图并实际行动”的一对一陪玩体验。黑客松阶段的重点不是覆盖所有游戏，而是用可运行、可复现、可现场演示的闭环证明方向成立。

## 1.2 黑客松核心命题

> **核心命题：** 玩家向 AI 队友表达一个游戏意图后，系统能够完成“输入 → 理解 → 决策 → 游戏行动 → 状态反馈 → 回应”的闭环；第二层证明该闭环可以实时打断并迁移到第二款游戏；第三层通过专用本地模型或实体桌宠形成技术 / 体验记忆点；第四层进一步验证自动评测、从轨迹学习技能和无接口视觉迁移能力。

## 1.3 目标用户与场景

| 用户 | 核心需求 |
|------|----------|
| 缺少固定游戏搭子的休闲玩家 | 需要一个随时在线、能接受指令、能一起完成简单合作目标的队友。 |
| 内容创作者 / 主播 | 希望 AI 队友能产生自然互动、状态反馈和可传播的节目效果。 |
| Agent / 游戏开发者 | 希望验证同一 Agent 编排层能否通过适配器进入不同游戏，并可量化比较不同决策模型。 |

## 1.4 非目标

- 不做竞技游戏自动化，不做任何绕过反作弊或破坏公平性的能力。
- 不在黑客松阶段承诺正式商业化指标、7×24 小时服务等级或全平台覆盖。
- 不以“能打所有游戏”为目标；第二层仅用第二款游戏证明架构可迁移。
- 不从零预训练通用基础模型；第三层仅允许基于人工整理、规则生成或已有游戏轨迹做小规模行为克隆、监督微调或策略蒸馏。
- 不让视觉模型或生成技能直接绕过安全与 Adapter 层控制游戏。

# 2. 四层交付结构总览

| 层级 | 定位 | 包含内容 | 唯一成功标准 | 失败影响 | 基准投入 |
|------|------|----------|--------------|----------|----------|
| 第一层 | 保底 / Must Have | AIRI 部署、Minecraft 接入、输入到行动再到反馈的完整链路 | 现场可连续完成预设 Minecraft 演示脚本 | 未通过则项目不可交付 | 50% |
| 第二层 | 高收益 / Should Have | 全双工实时语音；统一 Game Adapter；DST 第二游戏 | 玩家可打断 AI；同一编排层驱动两款游戏 | 未通过则竞争力下降，但不影响保底交付 | 30% |
| 第三层 | 超高收益 / Stretch | 游戏专用 SLM/SSM；实体硬件桌宠；软件形象 / 看板仅作配套降级 | 至少完成模型或硬件中的一条稳定主线，并可独立关闭 | 失败不影响第一、二层，仅损失技术壁垒或现场记忆点 | 12% |
| 第四层 | 前沿研究 / Optional | Agent Arena 自动评测；Replay-to-Skill；视觉影子世界模型 | 至少一个研究增强形成可重复证据或受控 Demo | 完全不影响前三层交付判断 | 8% |

## 2.1 层级门禁（Stage Gate）

1. Gate 1：AIRI 与 Minecraft 可稳定运行，完成端到端演示。通过后冻结第一层，不再继续美化。
2. Gate 2A：全双工语音的打断、取消和重规划可现场复现。
3. Gate 2B：DST 适配器复用同一上层意图与任务编排接口，而非复制一套独立 Agent。
4. Gate 3A（模型轨道）：SLM/SSM 在冻结测试集上完成动作正确率与延迟测试，并可一键回退 AIRI。
5. Gate 3B（硬件轨道）：实体桌宠能稳定消费体验事件；断电、断网或进程退出后主链路继续运行。
6. Gate 4A（评测轨道）：Agent Arena 可自动初始化标准任务、多轮运行并输出报告。
7. Gate 4B（学习 / 视觉轨道）：Replay-to-Skill 或 Shadow Observer 至少一项形成受控、可重复的 Demo。

## 2.2 推荐决策路径

| 当前状态 | 下一步决策 |
|----------|------------|
| Gate 1 未通过 | 只修 AIRI + Minecraft 阻断问题；停止所有其他开发。 |
| Gate 1 通过、Gate 2 未通过 | 集中完成全双工或 DST 中成功概率更高的一项，同时保持第一层冻结。 |
| Gate 2 可演示但无剩余时间 | 停止扩展，完成录屏、测试报告和现场 Runbook。 |
| Gate 2 稳定且有模型负责人 | 进入第三层模型轨道：用冻结小数据集训练 SLM/SSM，并建立最小离线基准。 |
| Gate 2 稳定且有硬件 / 体验负责人 | 进入第三层硬件轨道：先做 Experience Event Mock，再接入实体桌宠。 |
| 第三层至少一条主线稳定且仍有研究资源 | 进入第四层：优先 Agent Arena；Replay-to-Skill 与视觉影子模型按成功概率择一。 |

# 3. 总体技术架构与演进方式

## 3.1 第一层架构：直接复用 AIRI 现有链路

| 玩家输入 | AIRI 核心 | Minecraft 服务 |
|----------|-----------|----------------|
| AIRI 现有输入方式（文本 / 现有语音能力） | 模型配置、人格与 Agent 编排；不做通用化重构 | 连接 Minecraft 服务器；读取状态并执行动作 |
| 玩家意图 | 计划 / 指令 | 移动、跟随、采集、交互等 |
| 玩家可见回应 | 游戏事件反馈回 AIRI | 游戏世界状态 |

> **第一层架构原则：** 不先设计“完美的通用框架”。先按 AIRI 当前主仓库和 services/minecraft 的真实组织方式跑通；所有抽象工作延后到第二层。

## 3.2 第二层架构：实时语音与可迁移适配层

| 实时语音层 | Agent / 任务编排层 | 统一 Game Adapter | 游戏实现 |
|------------|---------------------|-------------------|----------|
| VAD、流式 STT/TTS、Barge-in、回声处理 | 意图解析、任务状态机、动作选择、对话策略 | observe / act / events / capabilities / health | Minecraft Adapter、DST Adapter |
| 并行对话通道 | 统一高层意图 | 标准状态与动作语义 | 各游戏本地 API / Mod / Bot |
| 可打断 / 可恢复 | 同一任务逻辑跨游戏复用 | 能力声明与降级 | 游戏内实际行动 |

## 3.3 统一适配契约（第二层新增）

| 接口 | 职责 |
|------|------|
| observe() | 返回角色、附近实体、资源、危险、任务进度等标准化状态。 |
| act(action) | 接收 follow / move / collect / interact / stop 等高层动作，并映射为游戏内操作。 |
| events() | 输出受击、死亡、资源获得、目标完成、玩家指令等事件流。 |
| capabilities() | 声明当前游戏支持的动作，避免 Agent 规划不可执行能力。 |
| health() | 报告连接、延迟、错误和可恢复状态，支持演示时快速降级。 |

## 3.4 第三层架构：专用模型与具身硬件双轨

| 模型轨道输入 | 专用行动模型 | 具身事件层 | 实体输出端 |
|--------------|--------------|------------|------------|
| 人工整理 / 规则生成 / 已有轨迹；状态、指令、动作标签 | Transformer SLM 或 SSM；动作、参数、置信度 | Experience Event Schema；speaking / action / danger / success | ESP32 / Raspberry Pi 桌宠；屏幕、LED、舵机或扬声器 |
| 冻结训练 / 验证 / 测试集 | 低置信度回退 AIRI | 只读订阅、节流和优先级 | 断线、断电不影响主链路 |

> **第三层架构原则：** SLM/SSM 与实体桌宠是并列、可独立交付的两条轨道。模型不依赖第四层 Agent Arena 才能训练；桌宠不依赖模型才能响应。任何轨道关闭后，第一、二层继续运行。

## 3.5 第四层架构：自动评测、技能学习与视觉迁移

| 受控场景与轨迹 | Agent Arena | 研究增强 | 安全边界与回流 |
|----------------|-------------|----------|----------------|
| Minecraft / DST 世界快照；状态、动作、事件、结果 | 任务初始化、自动复位；多轮执行、指标报告 | Replay-to-Skill；Shadow Observer | 统一 Game Adapter / 只读事件；人工确认、超时、取消 |
| 可选使用第三层模型 | 可评测 AIRI 或 SLM/SSM | 技能复用 / 无接口观察 | 失败不影响前三层 |

> **第四层架构原则：** 第四层是研究型加速器，不是第三层前置条件。Arena 可为 SLM/SSM 提供更可靠的数据与 A/B 测试；Replay-to-Skill 和视觉观察器必须分别经过 Adapter 安全执行或只读事件边界。

# 4. 第一层：Project AIRI 全链路保底

> **目标：** 不开发新产品功能，仅将 Project AIRI 的现有模块部署、配置并连接到 Minecraft，形成可重复演示的完整闭环。

## 4.1 第一层范围

- 固定 AIRI 版本或 commit，完成主仓库依赖安装和可复现启动。
- 配置可用的 LLM / 语音等 AIRI 现有依赖；以“能运行”为标准，不进行新模型训练。
- 启动 AIRI 的 Minecraft 服务，使 Bot 加入指定 Minecraft 服务器。
- 让玩家指令进入 AIRI，经过理解与决策后转为 Minecraft 行动。
- 让游戏状态或事件回流到 AIRI，并产生玩家可感知的回应。
- 固化一个短、稳定、可重复的现场演示脚本。

## 4.2 第一层明确不做

- 不实现全双工、持续监听、随时打断或语音并行处理。
- 不接入 Don't Starve Together 或任何第二款游戏。
- 不抽象通用 Game Adapter，不重构 AIRI 核心架构。
- 不建设 Agent Arena，不训练 SLM/SSM，不做 Replay-to-Skill 或视觉世界模型。
- 不增加长期记忆、情感识别、语音克隆、自定义人格编辑器。
- 不增加实体硬件、直播推流、弹幕互动、移动端或主机支持。
- 不承诺动作游戏级 100ms 延迟或正式产品 SLA。

## 4.3 第一层工作项与验收

| ID | 需求 / 工作项 | 实现说明 | 交付物 | 验收标准 |
|----|---------------|----------|--------|----------|
| L1-01 | 固定与启动 AIRI | 锁定版本、安装依赖、准备环境变量与模型提供商配置。 | 版本清单；环境模板；启动命令。 | 目标机器可从冷启动进入 AIRI 界面 / 服务。 |
| L1-02 | 启动 Minecraft 环境 | 准备受控服务器 / 世界，确认游戏版本与 Bot 兼容。 | 服务器配置；测试世界；账号权限说明。 | 真人玩家与 AIRI Bot 可进入同一世界。 |
| L1-03 | 连接 AIRI Minecraft 服务 | 按 AIRI 现有 services/minecraft 路径启动并连通核心。 | 服务启动脚本；连接日志。 | AIRI 能识别 Bot 在线并收到基本游戏状态。 |
| L1-04 | 打通指令到行动 | 使用 AIRI 原生输入方式发送简单意图，不新增输入产品形态。 | 预设指令集；演示录屏。 | Bot 执行跟随、停止、移动或简单交互中的至少三类。 |
| L1-05 | 打通反馈到回应 | 将成功、失败或环境变化回传到 AIRI。 | 事件日志；用户可见回应。 | 至少一种成功事件和一种失败事件触发合理回应。 |
| L1-06 | 固化演示脚本 | 限定地图、任务和指令，减少开放世界随机性。 | 3-5 分钟 Demo Runbook。 | 连续三次按脚本完成，无阻断级崩溃。 |
| L1-07 | 可复现交付 | 整理 README、配置模板、已知问题与快速重启方法。 | 交付包与故障排查清单。 | 非核心开发者可按文档重启演示。 |

## 4.4 第一层 Definition of Done

- AIRI、模型服务、Minecraft 服务和游戏服务器的版本均已锁定。
- 玩家输入能触发 Bot 行动，Bot 行动结果能回流并形成回应。
- 3-5 分钟演示脚本连续成功 3 次。
- 出现失败时，可在 2 分钟内按 Runbook 恢复到可演示状态。
- 第一层代码被冻结；后续修改不得破坏该基线。

# 5. 第二层：高收益增强

> **目标：** 在第一层稳定基线上增加两项最能证明产品价值的增强：真人式全双工实时语音；用 Don't Starve Together 证明游戏适配架构可迁移。

## 5.1 增强项 A：全双工实时语音

这里的“全双工”不是简单的语音输入输出，而是玩家和 AI 可同时说话，玩家能在 AI 发言或执行任务时打断，系统能取消旧响应、更新任务状态并给出新的行动。

| ID | 需求 / 工作项 | 实现说明 | 交付物 | 验收标准 |
|----|---------------|----------|--------|----------|
| V2-01 | 持续音频输入 | 接入麦克风持续流、VAD 与语音活动状态。 | 音频流状态面板 / 日志。 | 连续对话中可区分说话、静音和噪声。 |
| V2-02 | 流式识别与合成 | 流式 STT 提供增量文本；TTS 支持分段播放。 | 流式管道实现。 | 无需等待完整句子结束即可开始理解与回应。 |
| V2-03 | Barge-in 打断 | 玩家开口时停止当前 TTS，并向 Agent 发送中断事件。 | 打断控制器。 | 20 次测试中至少 18 次正确停止旧语音。 |
| V2-04 | 动作取消与重规划 | 语音打断同时影响正在执行的游戏任务。 | 任务取消令牌；状态机。 | “等等 / 停下 / 回来”能停止旧动作并执行新指令。 |
| V2-05 | 回声与串音处理 | 避免 AI 自己的 TTS 被再次识别为玩家输入。 | AEC / 输出抑制策略。 | 连续 10 分钟无明显自激对话。 |
| V2-06 | 延迟可观测 | 记录 VAD、STT、LLM、TTS、行动各阶段时间。 | 延迟看板或结构化日志。 | 能定位一次慢响应发生在哪个阶段。 |

## 5.2 增强项 B：Don't Starve Together 第二游戏

DST 的价值不是“再多做一款游戏”，而是验证同一 Agent / 语音 / 任务编排层能否通过新的适配器进入另一种状态模型和操作机制。优先采用受控专用服务器与服务端 Mod / 本地桥接方式，避免依赖屏幕视觉和通用键鼠自动化。

| ID | 需求 / 工作项 | 实现说明 | 交付物 | 验收标准 |
|----|---------------|----------|--------|----------|
| G2-01 | 定义通用状态模型 | 抽取角色位置、生命 / 饥饿 / 精神、附近实体、资源、危险与任务状态。 | GameState Schema。 | Minecraft 与 DST 映射到同一顶层状态结构。 |
| G2-02 | 定义通用动作语义 | 先覆盖 follow、move、collect、interact、stop、say 等最小动作。 | Action Schema。 | 上层 Agent 不需要知道底层按键或 Mod API。 |
| G2-03 | 实现 Minecraft Adapter | 将第一层已有链路包裹进统一契约，不改变其可运行基线。 | Minecraft Adapter。 | 第一层演示仍可通过回归测试。 |
| G2-04 | 实现 DST Bridge | 通过专用服务器 Mod / 本地桥接读取状态、发送动作与事件。 | DST Mod / Bridge。 | AI 角色可进入受控 DST 世界并执行最小动作集。 |
| G2-05 | 实现 DST Adapter | 将 DST 状态和动作映射到统一契约。 | DST Adapter。 | 同一意图“跟随 / 停止 / 收集”可在 DST 执行。 |
| G2-06 | 跨游戏回归 | 使用同一组高层意图测试两款游戏。 | 跨游戏测试报告。 | 核心编排逻辑不复制；仅 Adapter / Bridge 不同。 |

## 5.3 第二层建议补充：任务可靠性与降级

- 为每个动作增加超时、重试与 cancel，防止 Agent 长时间卡在不可达目标。
- 适配器通过 capabilities() 声明能力；不支持的动作应立即返回可解释失败。
- 演示任务使用小型状态机或技能函数承接执行，LLM 负责意图与高层规划，避免逐帧控制。
- 保留文本输入作为语音故障的现场降级通道，但不把它作为新的用户功能宣传。

## 5.4 第二层 Definition of Done

- 玩家可在 AI 说话时打断，旧语音在可感知范围内停止。
- 玩家可在 AI 执行动作时用语音停止或改派任务。
- Minecraft 和 DST 均通过统一 Game Adapter 契约运行。
- 两款游戏至少共享同一组 follow / stop / collect / interact 高层意图。
- 任一第二层模块关闭后，第一层仍能独立演示。

# 6. 第三层：超高收益可选增强

> **目标：** 在第二层稳定后，选择一条或两条最能提升黑客松竞争力的增强主线：用自训练 Transformer SLM / SSM 建立低延迟专用行动模型；用实体硬件桌宠建立具身陪伴和现场记忆点。两条轨道互不依赖，均可一键关闭。

## 6.1 第三层定位与双轨选择

| 轨道 | 核心价值 | 最小实现 | 关键依赖 | 推荐条件 |
|------|----------|----------|----------|----------|
| T3-A｜游戏专用 SLM / SSM | 形成自有模型资产；降低高频动作决策延迟与云调用。 | 至少 5 类高层动作；冻结小数据集；本地推理与置信度回退。 | Game Adapter；轨迹 / 合成样本；目标推理硬件。 | 团队有模型训练与部署能力。 |
| T3-B｜实体硬件桌宠 | 形成具身陪伴、情绪反馈与强烈现场记忆点。 | 小屏 / LED / 舵机 / 扬声器的最小组合；至少 5 类事件反馈。 | Experience Event；独立通信进程；硬件。 | 团队有嵌入式或硬件快速原型能力。 |
| 配套 Backlog｜软件降级 | 硬件未到场或故障时保留可见体验。 | 浏览器状态卡或简易 Avatar 复用同一事件协议。 | Experience Event。 | 仅在不影响主线时实现。 |

- 两条轨道可以并行，也可以只选择成功概率最高的一条；第三层不要求模型与硬件同时完成。
- SLM/SSM 可以使用人工整理、规则系统生成、教师模型标注或第一 / 二层已有日志构建数据集，不等待第四层 Arena。
- 实体桌宠只消费结构化体验事件，不读取模型内部思维链，也不向游戏控制链路写入指令。
- Live2D / VRM、观众看板、人设包与局内记忆不再作为独立交付项，仅作为桌宠的软件备份或配套 Backlog。

## 6.2 游戏专用本地小模型：Transformer SLM / SSM

SLM 表示小语言 / 行动模型的规模与用途；Transformer 与 SSM 表示可选架构。黑客松阶段不从零预训练通用模型，而是将游戏状态、玩家指令与目标动作构造成小规模监督数据，训练一个本地“反射层”。开放式聊天和复杂规划仍由 AIRI 处理。

| ID | 需求 / 工作项 | 实现说明 | 交付物 | 验收标准 |
|----|---------------|----------|--------|----------|
| M3-01 | 动作边界 | 限定 follow、stop、collect、interact、avoid、return 等高频高层动作。 | 动作标签集。 | 至少支持 5 类动作，不承担开放式聊天。 |
| M3-02 | 最小数据集 | 整合人工标注、规则生成、教师模型标注或已有轨迹。 | Dataset Manifest。 | 数据来源、版本和许可清晰；训练 / 测试无重复。 |
| M3-03 | 架构与训练 | 选择轻量 Transformer SLM 或 SSM，进行行为克隆、监督微调或策略蒸馏。 | 训练脚本、配置、权重。 | 目标机器 / 单卡可完成训练或加载既有训练结果。 |
| M3-04 | 离线基准 | 使用冻结测试集比较动作正确率、推理延迟和错误类型。 | Offline Evaluation Report。 | 动作选择正确率目标 ≥90%；报告可复现。 |
| M3-05 | 置信度路由 | 输出动作、参数和置信度；未知任务或低置信度回退 AIRI。 | Router Service。 | 关闭模型、超时或低置信度时主链路继续。 |
| M3-06 | 在线受控验证 | 在固定 Demo 场景人工重复测试成功率与延迟。 | 对比记录 / 简易看板。 | 决策延迟目标降低 ≥50%，成功率不明显下降。 |

## 6.3 实体硬件桌宠

桌宠是第三层的具身体验主线：它把 AI 的说话、思考、行动、危险、受击和成功转化为实体表情、灯光、动作或声音。目标不是制作完整消费电子产品，而是用最小硬件组合证明 AI 队友可以离开屏幕、形成持续陪伴感。

| ID | 需求 / 工作项 | 实现说明 | 交付物 | 验收标准 |
|----|---------------|----------|--------|----------|
| H3-01 | 硬件形态 | 选择 ESP32 / Raspberry Pi，并组合小屏、LED、舵机或扬声器中的必要部件。 | 可运行桌宠原型。 | 上电后可独立进入待机状态。 |
| H3-02 | 统一体验事件 | 定义 speaking、thinking、action、success、danger、hurt、death 等事件及优先级。 | Experience Event Schema。 | Mock、桌宠和软件备份使用同一协议。 |
| H3-03 | 通信与状态机 | 通过 WebSocket / MQTT / 串口消费事件，处理节流、超时与状态覆盖。 | Hardware Bridge。 | 连续事件下无明显卡死或状态混乱。 |
| H3-04 | 表情与动作映射 | 将事件映射为屏幕表情、灯光、摆动、口型或音效。 | 映射表与素材。 | 至少 5 类状态肉眼可区分。 |
| H3-05 | 语音同步 | AI 说话时同步口型 / 灯效，或播放同一路 TTS。 | 同步模块。 | 音频开始 / 结束与状态基本一致。 |
| H3-06 | 故障隔离与降级 | 硬件独立进程运行；支持断线、重连和浏览器状态卡降级。 | 故障处理 Runbook。 | 拔电、断网或关闭进程后主 Demo 继续。 |

## 6.4 第三层 Definition of Done 与取舍规则

- 至少完成模型轨道或硬件轨道中的一条；两条都完成属于加分，不是交付前提。
- 模型轨道必须有冻结测试集、可复现结果和低置信度回退；不得只展示训练 loss。
- 硬件轨道必须稳定消费至少 5 类事件，并通过断电 / 断线隔离测试。
- 第三层任何增强关闭后，第一、二层仍可按原脚本运行。
- 若模型没有速度或稳定性收益，不进入主 Demo；若硬件不稳定，切换软件状态卡或录屏。
- 第三层不依赖第四层，但可在第四层 Agent Arena 完成后补充更可靠的自动评测。

# 7. 第四层：前沿研究型增强

> **目标：** 探索系统能否从一次性 Demo 进化为可自动测量、可从经验沉淀技能、并能在缺少结构化接口时继续感知的 Agent 平台。第四层只包含 Agent Arena、Replay-to-Skill 和视觉影子世界模型，优先级低于第三层。

## 7.1 第四层模块与依赖关系

| 模块 | 核心价值 | 最小实现 | 与第三层关系 | 边界 |
|------|----------|----------|--------------|------|
| T4-0｜Agent Arena 自动评测 | 建立可重复基线、轨迹数据和回归报告。 | 3 个标准任务；自动复位；每项 10 轮；指标报告。 | 可评测 AIRI 与第三层 SLM/SSM，但不是其前置。 | 不成为现场主链路依赖。 |
| T4-1｜Replay-to-Skill | 把成功经验沉淀为可复用技能。 | 一条成功轨迹生成一个可读、可取消的技能。 | 可使用 Arena 轨迹，也可使用人工录制轨迹。 | 技能必须经过 Adapter / 状态机执行。 |
| T4-2｜视觉影子世界模型 | 验证没有 API / Mod 时的只读感知迁移路径。 | 固定窗口识别至少 3 类事件。 | 可为桌宠 / 语音 / Arena 提供补充事件。 | 不直接控制角色。 |

## 7.2 Agent Arena：自动评测与轨迹基础

Agent Arena 自动初始化受控场景、下发任务、多轮运行、记录轨迹并生成报告。它既能评测 AIRI 基线，也能在第三层 SLM/SSM 完成后做更可靠的 A/B 对比，但 Arena 失败不得影响第三层模型的最小离线验证。

| ID | 需求 / 工作项 | 实现说明 | 交付物 | 验收标准 |
|----|---------------|----------|--------|----------|
| A4-01 | 标准任务集 | 建立 follow、collect、stop / danger 等至少三个可重复场景。 | 任务定义与世界快照。 | 每个任务可脚本初始化并复位。 |
| A4-02 | 统一轨迹记录 | 记录状态、指令、动作、事件、结果与各阶段延迟。 | Trajectory Schema。 | 单次运行可回放关键决策证据。 |
| A4-03 | 自动多轮执行 | 同一场景连续运行，成功或失败后自动复位。 | Arena Runner。 | 每个场景可自动执行 10 轮。 |
| A4-04 | 指标与失败归因 | 输出成功率、耗时、动作数、调用数、延迟和失败原因。 | JSON / CSV 报告。 | 一次命令生成可比较报告。 |
| A4-05 | 数据划分与版本 | 训练、验证、测试按场景 / 种子隔离，冻结版本。 | Dataset Manifest。 | 避免轨迹泄漏，结果可追溯。 |
| A4-06 | 基线与回归 | 保存 AIRI 基线；可选加入第三层模型对比。 | Baseline / A-B Report。 | 同一版本、同一任务可重复比较。 |

## 7.3 Replay-to-Skill：从成功轨迹生成可复用技能

系统将一条成功轨迹抽象为参数化技能，例如“找到树木 → 采集木材 → 回到玩家附近 → 报告完成”。技能不是任意底层按键宏，而是具有前置条件、成功条件、超时、取消和失败分支的行为树或状态机。

| ID | 需求 / 工作项 | 实现说明 | 交付物 | 验收标准 |
|----|---------------|----------|--------|----------|
| S4-01 | 成功轨迹输入 | 使用 Arena 轨迹或人工录制且已确认成功的轨迹。 | 候选轨迹列表。 | 失败 / 异常轨迹不得自动发布。 |
| S4-02 | 轨迹抽象 | 将坐标与具体对象抽象为目标、条件、动作和终止条件。 | Skill IR。 | 技能可在不同初始位置复用。 |
| S4-03 | 技能生成 | 输出行为树、有限状态机或参数化 Skill Definition。 | 可读技能文件。 | 包含进入、成功、超时、失败和取消分支。 |
| S4-04 | 人工确认与沙箱 | 新技能先在受控场景测试，经确认后进入技能库。 | Review Gate。 | 未经确认不能用于主世界。 |
| S4-05 | 安全执行 | 通过统一 Adapter 执行，支持 cancel、timeout 和回退。 | Skill Runner。 | 玩家“停止”可中断技能。 |
| S4-06 | 重复性验证 | 在不同初始位置运行生成技能。 | 复现报告。 | 连续 3 次至少成功 2 次。 |

## 7.4 视觉影子世界模型（Shadow Observer）

视觉观察器与结构化 Adapter 并行，只识别和补充事件，不直接 act。其目的不是宣称完成通用纯视觉游戏 Agent，而是证明架构可以逐步进入缺少 API / Mod 的游戏。

| ID | 需求 / 工作项 | 实现说明 | 交付物 | 验收标准 |
|----|---------------|----------|--------|----------|
| W4-01 | 画面采集 | 低帧率截图或视频流采集固定游戏窗口。 | Capture Service。 | 采集不明显影响游戏帧率。 |
| W4-02 | 事件定义 | 限定低血量、附近危险、目标资源 / UI 完成等至少三类事件。 | Vision Event Schema。 | 事件可进入统一事件总线。 |
| W4-03 | 视觉推理 | 使用现有视觉模型或轻量检测模型输出事件与置信度。 | Observer Service。 | 画面到事件目标延迟 ≤1 秒。 |
| W4-04 | 状态对照 | 与 Adapter 结构化状态比较一致性、漏检与误报。 | 对照日志。 | 展示至少一种补充或校验价值。 |
| W4-05 | 只读消费 | 视觉事件可触发语音提醒、桌宠反馈或 Arena 标记，但不直接控制。 | 只读事件消费者。 | 关闭视觉服务后任务执行不受影响。 |
| W4-06 | 固定场景评测 | 在预设场景测试准确率与误报。 | 视觉评测报告。 | 三类事件准确率目标 ≥80%。 |

## 7.5 第四层 Definition of Done 与停止条件

- 第四层不要求三项全部完成；至少一项形成可重复证据或受控 Demo 即可。
- 若选择 Agent Arena：至少 3 个标准任务、每项自动运行 10 轮并生成报告。
- 若选择 Replay-to-Skill：至少生成一个可读、可取消的技能，连续 3 次至少成功 2 次。
- 若选择视觉影子模型：至少识别 3 类固定事件，目标准确率 ≥80%、延迟 ≤1 秒。
- 所有模块关闭后，第一、二、三层继续运行；第四层不得成为现场演示的必需基础设施。
- 第一层或第二层出现回归、第三层主线尚未冻结、或剩余时间不足总周期 20% 时，第四层立即停工。

| DST / 适配层负责人 | 通用契约、DST Bridge、DST Adapter、跨游戏回归。 | Phase 4。 |
| 模型负责人 | 数据集、Transformer SLM / SSM、离线评测、推理路由与回退。 | Gate 2 后进入 Phase 5A。 |
| 硬件 / 体验负责人 | Experience Event、桌宠通信、表情动作、软件降级和演示 Runbook。 | Gate 1 后可做 Mock；Phase 5B 正式接入。 |
| 研究增强负责人 | Agent Arena、Replay-to-Skill 或视觉 Shadow Observer。 | 第三层主线冻结后进入 Phase 6。 |

## 8.3 并行与依赖规则

- 模型负责人可在第二层期间离线准备小数据集，但不得要求第一层为采集数据重构。
- 硬件负责人可提前使用 Mock Experience Event 开发；正式接入只读事件，不向 Agent 或游戏写入。
- 第三层 SLM/SSM 不能以第四层 Arena 为前置条件；先完成冻结测试集与人工受控验证。
- 第四层 Arena 完成后，可以补充第三层模型的自动 A/B 报告，但不能改变第三层是否可独立运行。
- Replay-to-Skill 可使用人工录制轨迹；视觉观察器可独立开发，但均必须遵守 Adapter / 只读事件边界。
- 每次跨层合并必须运行第一层回归脚本，并保留可快速回滚的 tag / 分支。

## 8.4 每日 / 每阶段决策规则

- 先修“演示阻断”，再修“体验瑕疵”，最后做“新功能”。
- 任何跨层修改必须跑第一层回归脚本。
- 每天至少保留一份可运行版本和一段成功录屏。
- 第三层只保留能稳定展示技术价值或具身记忆点的主线；第四层只保留有可重复证据的实验。
- 距最终演示不足 20% 总时间时，停止新增功能，只做稳定化和讲解材料。

# 9. 演示脚本与验收指标

## 9.1 第一层保底 Demo（3-5 分钟）

1. 启动 AIRI 与 Minecraft 服务，展示 Bot 在线并进入同一世界。
2. 玩家通过 AIRI 当前可用输入方式发出“跟我来 / 到这里 / 停止”等指令。
3. Bot 在 Minecraft 中执行至少三类基础行动。
4. 制造一次成功事件与一次失败 / 不可执行事件，展示状态回流和回应。
5. 以一个简单合作目标结束，例如到达指定点、收集指定资源或完成一次交互。

## 9.2 第二层竞争力 Demo（5-7 分钟）

1. 玩家与 AI 连续语音对话，AI 在回复期间玩家直接插话。
2. 玩家说“等等，停下，回来”，AI 停止旧语音和旧游戏动作并重规划。
3. 在 Minecraft 展示 follow / collect / stop。
4. 切换到 DST，使用同类高层指令完成跟随、采集或交互。
5. 展示适配器看板：同一 Agent 编排层，底部连接不同 Game Adapter。

## 9.3 第三层超高收益 Demo（3-5 分钟，可择一或组合）

- **SLM/SSM 主线**：在同一固定任务中切换 AIRI 基线与本地行动模型，展示动作选择、置信度、决策延迟和回退。
- **实体桌宠主线**：触发 speaking、thinking、action、danger、success、death 等事件，展示屏幕、灯光、动作或声音同步变化。
- **组合主线**：本地模型执行高频动作时，桌宠同步显示“思考 → 行动 → 成功 / 失败”，形成模型能力与具身体验的统一叙事。
- 现场断开模型服务或给桌宠断电，证明主游戏与语音链路继续运行。
- 只展示结构化状态、动作、置信度和结果，不展示模型内部思维链。

## 9.4 第四层研究增强 Demo（4-6 分钟，择一主线）

- **Agent Arena 主线**：自动初始化同一任务、多轮运行并生成 AIRI / 可选 SLM 的成功率、延迟和失败归因报告。
- **Replay-to-Skill 主线**：录制或选取一次成功轨迹，生成可读技能，并在不同初始位置重复执行。
- **视觉主线**：关闭或遮蔽部分结构化事件，让 Shadow Observer 从画面识别危险、低血量或目标资源，并触发只读提醒 / 桌宠反馈。
- 研究模块失败时立刻关闭，回到前三层主 Demo，不在现场调试研究管线。

## 9.5 分层指标

| 层级 | 指标 | 验收定义 | 级别 |
|------|------|----------|------|
| 第一层 | 链路完整性 | 输入、理解、行动、反馈、回应五段均有可观察证据。 | 必须 |
| 第一层 | 演示稳定性 | 预设脚本连续成功 3 次；无阻断级崩溃。 | 必须 |
| 第一层 | 恢复能力 | 常见故障可在 2 分钟内按 Runbook 恢复。 | 必须 |
| 第二层 | 语音打断成功率 | 20 次人工测试中 ≥18 次停止旧 TTS 并接收新指令。 | 目标 |
| 第二层 | 动作取消成功率 | 10 次“停止 / 改派”中 ≥8 次正确取消旧任务。 | 目标 |
| 第二层 | 跨游戏复用 | 两款游戏共享上层意图、任务状态机和 Agent 编排。 | 必须 |
| 第三层 | SLM/SSM 价值 | 测试集动作正确率目标 ≥90%；决策延迟目标降低 ≥50%，成功率不明显下降。 | 可选 |
| 第三层 | 桌宠可见性 | 至少 5 类事件肉眼可区分；事件后目标 1 秒内感知变化。 | 可选 |
| 第三层 | 故障隔离 | 模型服务或硬件关闭 / 断开后第一、二层继续运行。 | 必须 |
| 第四层 | Arena 可重复性 | 至少 3 个任务，每个任务自动运行 10 轮并生成报告。 | 可选 |
| 第四层 | Replay 技能复现 | 生成技能连续 3 次至少成功 2 次，且支持取消。 | 可选 |
| 第四层 | 视觉事件识别 | 固定场景三类事件准确率目标 ≥80%，延迟目标 ≤1 秒。 | 可选 |
| 第四层 | 研究隔离 | 任一研究模块关闭后前三层继续运行。 | 必须 |

# 10. 风险、降级与停止条件

## 10.1 分层风险与现场降级

| 风险 | 影响 | 预防 | 现场降级 |
|------|------|------|----------|
| AIRI 版本 / 依赖漂移 | 无法启动或接口变化 | 固定 commit 与 lockfile；保存可运行镜像 / 安装缓存。 | 回退已验证版本，不追最新 main。 |
| Minecraft 协议或账号问题 | Bot 无法入服 | 固定游戏版本；准备受控服务器与专用账号。 | 切换本地演示服务器。 |
| LLM 行动不稳定 | 开放任务失败 | 缩小地图和任务；关键动作由技能函数 / 状态机执行。 | 只演示高成功率预设任务。 |
| 语音回声或打断失效 | AI 自说自话、体验混乱 | 输出抑制、AEC、打断控制器、日志定位。 | 关闭全双工，回退第一层输入方式。 |
| DST Mod / Bridge 不稳定 | 第二游戏无法演示 | 受控专服；最小动作集；健康检查。 | 展示 Adapter Mock 和 Minecraft 回归。 |
| 训练数据不足或泄漏 | 小模型过拟合、指标失真 | 冻结数据版本；按场景划分；独立测试集。 | 关闭模型路由，只展示 AIRI 基线。 |
| SLM/SSM 无速度 / 稳定性收益 | 增加复杂度但无价值 | 先测推理预算；限制动作与上下文；置信度回退。 | 从主 Demo 移除模型，保留训练结果说明。 |
| 硬件桌宠故障 | 无响应、状态错乱或拖垮链路 | 独立进程、超时、断线隔离、软件状态卡备份。 | 断开硬件，切换浏览器状态卡 / 录屏。 |
| Arena 场景不可复位 | 实验结果不可比较 | 固定世界快照 / 种子；脚本化复位。 | 改为离线轨迹回放和报告。 |
| 生成技能不安全 / 不稳定 | 错误动作污染主世界 | 人工确认、沙箱、超时、取消和 Adapter 限制。 | 只展示技能文件或离线回放。 |
| 视觉误报 / 幻觉 | 错误提醒或错误行动 | 只读 Shadow Observer；置信度阈值；结构化状态对照。 | 关闭视觉事件消费者。 |
| 范围膨胀 | 核心链路未完成 | 严格 Gate；新增想法只进第三 / 四层 Backlog。 | 冻结功能并转入稳定化。 |

## 10.2 全局停止条件

- 第一层未通过 Gate 1：停止所有第二、三、四层开发。
- 第一层回归被高层修改破坏：立即回滚，不在现场版本上继续调试。
- 第二层尚不能稳定演示：停止第三、四层正式接入，只保留独立 Mock。
- 第三层模型没有冻结测试结果或硬件未通过断线隔离：不进入主 Demo。
- 第四层 Arena 尚无可靠基线或数据泄漏未解决：禁止宣称模型优于基线。
- 第四层 Replay / 视觉模块需要绕过 Adapter 或直接控制角色：取消接入。
- 新功能无法在剩余时间内形成可演示闭环：删除或降级为讲解材料。
- 现场网络或云模型不可控：切换预先准备的本地 / 备用服务，或使用成功录屏兜底。

## 11.1 术语

| 术语 | 定义 |
|------|------|

| 全链路 | 玩家输入 → Agent 理解 / 决策 → 游戏行动 → 游戏状态 / 事件反馈 → 玩家可见回应。 |
| 全双工实时语音 | 双方可同时说话；玩家可打断 AI，系统能停止旧语音并取消 / 重规划游戏任务。 |
| Game Adapter | 把上层统一状态与动作契约映射到具体游戏实现的适配模块。 |
| Bridge | 位于游戏侧或本地进程中的连接层，例如 Minecraft Bot 服务或 DST 服务端 Mod。 |

| SLM | Small Language Model；面向特定任务、参数和推理成本较小的语言 / 行动模型。 |

| Replay-to-Skill | 把成功游戏轨迹抽象为可复用行为树、状态机或参数化技能的流程。 |
| Shadow Observer | 只读视觉观察器；从画面补充事件，但不直接执行游戏动作。 |
| Experience Event | 供桌宠、Avatar、看板等消费的统一体验事件，例如 speaking、danger、success。 |
| Stage Gate | 阶段门禁；上一层达到明确验收标准后，才允许进入下一层。 |

- 实体硬件桌宠由第四层前移到第三层，并新增统一 Experience Event、硬件状态机、断线隔离和软件降级要求。

- 第三层 SLM / SSM 不再强依赖 Agent Arena；可用冻结小数据集完成最小训练与离线验证，Arena 仅作为后续自动评测升级。
- Live2D / VRM、观众看板、人设包和局内短期记忆不再作为独立层级目标，仅保留为第三层桌宠的配套 Backlog。
